{{JSRef}}

The getTimezoneOffset() method returns the difference, in minutes, between a date as evaluated in the UTC time zone, and the same date as evaluated in the local time zone.

{{EmbedInteractiveExample("pages/js/date-gettimezoneoffset.html")}}

Syntax

date.getTimezoneOffset()

Return value

The difference, in minutes, between date, as evaluated in the UTC time zone, and as evaluated in the local time zone.

Description

date.getTimezoneOffset() returns the difference, in minutes, between date as evaluated in the UTC time zone, and date as evaluated in the local time zone — that is, the time zone of the host system in which the browser is being used (if the code is run from the Web in a browser), or otherwise the host system of whatever JavaScript runtime (for example, a Node.js environment) the code is executed in.

Negative values and positive values

The number of minutes returned by getTimezoneOffset() is positive if the local time zone is behind UTC, and negative if the local time zone is ahead of UTC. For example, for UTC+10, -600 will be returned.

Varied results in Daylight Saving Time (DST) time zones

In a time zone that annually shifts in and out of Daylight Saving Time (DST), the number of minutes returned by calling getTimezoneOffset() can vary. Here’s an example:

// On a host system running in the 'Europe/Berlin' time zone

const dateBeforeDST = new Date('2021-03-01');
dateBeforeDST.toISOString();                      // Returns '2021-03-01T00:00:00.000Z'
dateBeforeDST.getTimezoneOffset();                // Returns -60
Intl.DateTimeFormat().resolvedOptions().timeZone; // Returns 'Europe/Berlin'

const dateDuringDST = new Date('2021-05-01');
dateDuringDST.toISOString();                      // Returns '2021-05-01T00:00:00.000Z'
dateDuringDST.getTimezoneOffset();                // Returns -120
Intl.DateTimeFormat().resolvedOptions().timeZone; // Returns 'Europe/Berlin'

From that example, here are some key points:

  • The time zone doesn’t change; it’s Europe/Berlin regardless of the date being evaluated.
  • The created Date objects (dateBeforeDST and dateDuringDST) carry no time zone offset.
  • The time zone offset is evaluated on the basis of what the local time was or will be on the given date: specifically, on the difference between what the time in the UTC time zone was or will be on the given date, and what the local time in the Europe/Berlin time zone was or will be on the given date.

In other words: The time zone offset comes from the environment in which the Date object is evaluated, not from the Date object itself. So, doing an evaluation in the Europe/Berlin time zone of a date that takes place before the period when the Europe/Berlin time zone has shifted into DST produces the time zone offset "-60", while doing an evaluation in the Europe/Berlin time zone of a date that takes place during the period when the Europe/Berlin time zone has shifted into DST produces the time zone offset "-120".

Contrast that Europe/Berlin time zone example with the following Asia/Tokyo example:

// On a host system running in the 'Asia/Tokyo' time zone

const dateBeforeDST = new Date('2021-03-01');
dateBeforeDST.toISOString();                      // Returns '2021-03-01T00:00:00.000Z'
dateBeforeDST.getTimezoneOffset();                // Returns -540
Intl.DateTimeFormat().resolvedOptions().timeZone; // Returns 'Asia/Tokyo'

const dateDuringDST = new Date('2021-05-01');
dateDuringDST.toISOString();                      // Returns '2021-05-01T00:00:00.000Z'
dateBeforeDST.getTimezoneOffset();                // Returns -540
Intl.DateTimeFormat().resolvedOptions().timeZone; // Returns 'Asia/Tokyo'

So in an Asia/Tokyo time zone environment, which doesn’t annually shift in and out of DST, calling getTimezoneOffset() will always return the same time zone offset, "-540", regardless of the date.

  • A time zone offset is simply a number indicating a difference in time relative to the UTC time zone. A time zone offset can be either a positive or negative number, and can be represented as a number of hours (as in, for example, GMT+0900) or — as is the case with getTimezoneOffset() — as a number of minutes (for example, -540).

Keep in mind, though, that the above explanations are a simplification. In implementations, the IANA time zone database is used to determine the effect of DST on the calculation of the time-zone difference.

And also keep in mind that getTimezoneOffset() doesn’t return a {{Glossary("time zone")}} — instead it returns a {{Glossary("time zone offset")}}. To determine the configured time zone, you can use the following code:

Intl.DateTimeFormat().resolvedOptions().timeZone
