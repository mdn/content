The location.reload() method reloads the current URL, like the Refresh button.

The reload may be blocked and a SECURITY_ERROR {{domxref("DOMException")}} thrown. This happens if the {{Glossary("origin")}} of the script calling location.reload() differs from the origin of the page that owns the {{domxref("Location")}} object. See Same-origin policy for more information.

Syntax

location.reload();

location.reload() has no parameter

Any parameter you specify in a location.reload() call will be ignored and have no effect of any kind.

You may come across instances of location.reload(true) in existing code that was written with the assumption that calling it with that boolean true parameter tells browsers to bypass the browser cache and force-reload the current document. However, no current browsers actually do anything different when given such a true/false “force-reload” parameter in a location.reload() call.

Despite the fact that calling location.reload(true) has exactly the same effect in browsers that calling location.reload() does, there has for years been extensive confusion about it, and there has for years been extensive cargo-cult usage of it.

Although the confusion seems to cleared up in recent years, a GitHub "location.reload(true)" search still returns several hundred thousand results. So there’s a lot of existing code which still has it.

How the confusion was ever able reach that point is unclear. But the history behind it all just goes back to the fact that once, long ago, some versions of Netscape Navigator did once do something different with location.reload(true). And so it made its way into code that later got copied or parroted into other code, and that kept going on long after Netscape Navigator was gone, and so then it ultimately ended up in code all over the place, without anybody apparently ever testing the code to see if it actually had any effect.

It’s worth noting, anyway, that even in those old Netscape Navigator versions, the parameter wasn’t strictly a “force-reload” parameter; instead it was more of a “force-GET” parameter. That’s because its actual purpose was with responses to POST requests initiated from forms: Calling location.reload(true) would cause the request to be re-sent — but as a GET request. So the actual purpose of the parameter was to avoid causing the browser to re-POST the form data it had already posted to the server.

Then, at one point after Netscape Navigator has already fallen out of use, the W3C Web APIs Working Group took up an issue to consider adding a true/false parameter to the specification for location.reload(). However, it was never actually added.

So a boolean parameter is not part of the current specification for location.reload() — and in fact has never been part of any specification for location.reload() ever published.
